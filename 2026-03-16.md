## Day 01 — A Day in the Life of a SOC Analyst

**Date:** March 16, 2026
**Pages covered:** 149 – 157
**Book:** Blue Team Handbook

---

### What this chapter is about

This chapter explains what a SOC Analyst actually does during a working shift. Before reading this, I had a vague idea that analysts just watch alerts. After reading it, I understand it is a much more structured role with specific tasks that must be done every single day. The chapter gives a repeatable framework that analysts follow to make sure nothing important gets missed.

---

### The 7 daily objectives of a SOC Analyst

The book lists seven things every analyst should cover during a shift:

1. **Alarm Triage** — Go through incoming alerts in priority order. If an alert looks real, the analyst collects some initial data, starts a ticket, investigates, or escalates it to someone more senior.

2. **Dashboard Review** — Look at the summary dashboards to get a broad picture of what is happening in the environment. This is about situational awareness, not deep investigation.

3. **Security State Review** — Make sure that the right data is actually arriving into the SIEM platform. If a data source stops sending logs, you would never know unless you check.

4. **SIEM Component Health Check** — The tools the SOC uses need to be healthy too. Check that the SIEM system itself is running properly, every day.

5. **Identify and Report Operational Issues** — SOC analysts often spot IT problems through log data before the IT team does. Reporting these makes the SOC a useful part of the whole organization, not just a security silo.

6. **Active Threat Hunting** — Going beyond alerts and proactively searching security data for signs of threats that no alert was triggered for.

7. **Security Intelligence Review** — Read vendor bulletins, threat feeds, and security news to stay current on what threats are active in the world right now.

---

### Figure 7 — Daily Analysis Overview

The book includes a diagram on page 150 showing how all these tasks connect. The way I understood it:

- Alarm triage, dashboard review, and the security state check all run in parallel at the start of a shift
- The security state check and threat intelligence both feed into the investigation process
- Everything feeds into a central investigation and documentation step
- Supporting data like asset inventories, network maps, and vulnerability scans sit alongside the whole process as reference material
- At the end of the shift, the analyst prepares a situational awareness update for whoever takes over next

The key note the book makes is: wherever possible, enrich your alerts and events with environmental data. An alert with context is much easier to make a decision on than a raw alert with nothing attached to it.

---

### Alarm Triage in detail

This is the most important task of the day. The analyst works through alarms from highest severity to lowest. The book says severity should follow a model  if the SIEM uses the Cyber Kill Chain, then System Compromise sits at the top because it means an attacker has already successfully breached something.

The triage process follows this decision path:

1. Review the prioritized alarm list
2. Triage and classify each alarm
3. Ask whether this is likely a true positive or a false positive
   - False positive: classify it and mark it resolved
   - True positive: investigate it, act on it, escalate if needed, then classify and close

**Analyst action table — Table 29 from the book**

The chapter includes a table of actions an analyst can take on any given alarm. I wrote it out here because it gave me a concrete picture of real decisions:

| Action | Real example from the book |
|--------|---------------------------|
| Close non-actionable alerts with a supporting reason | Scan from the internet against a host that is intentionally internet-facing |
| Close confirmed false positives | QuickTime alert on a machine that does not even have QuickTime installed |
| Mark for investigation if more research is needed | NIDS alert that needs to be cross-referenced against other data sources |
| Escalate if it is beyond your skill level | PowerShell exploit code combined with multiple authentication failures on the same host |
| Process directly if within your skill level | Potentially unwanted programs detected — generate a service desk ticket and mark under remediation |

### Dashboard and Summary Data Review

This section introduced me to a concept called Long Tail Analysis (LTA). The idea is to review data from the most common events down to the rare single occurrences. Those rare events are where interesting things sometimes hide. The objective is big-picture awareness making sure no important single event gets buried in the noise.

Other things to check during the dashboard review:
- Threat intelligence activity is any known malicious IP, domain, or file hash appearing in the environment right now?
- Event type changes in the SIEM have any new event types appeared, or have expected ones gone silent?
- Asset vulnerabilities that may have newly appeared

---

### Security State Data Review

This section covered something I had not considered before: data sources can silently stop reporting. If a firewall stops sending logs to the SIEM and nobody notices, you have a blind spot in your monitoring. The SOC needs a process to detect this.

Critical Device Review means checking outbound traffic, internal scan data, log volumes, event variety, account management activity, and vulnerability status for critical systems. NetFlow is mentioned as a useful tool for checking for unexpected new port or system combinations.

**Five methods the book gives for validating data health:**

1. Automate an alarm that fires if a data source's reporting profile changes significantly. Every asset that is supposed to send data should have an expected reporting attribute set.
2. Use scripts to check whether log files exist and whether they are growing. A simple file existence and growth check is easy to build for syslog sources.
3. Once a day, manually review the variety of data arriving from each source not just volume. If a VPN appliance suddenly stops sending VPN access events but keeps sending other data, that gap is worth investigating.
4. Create synthetic transactions — artificial test events that flow through the full pipeline and confirm the chain is working end to end.
5. Monitor event throughput for irregular patterns. Most environments have predictable rhythms — a spike at the start of the workday, lower volume on holidays. Volumetric changes are indicators, not proof of intrusion. That distinction matters.

### SOC Support System — Component Health Review

When analysts are not handling alerts, they should be doing maintenance on the SOC's own tools. The book specifies a daily health checklist:

- Disk space trends on SIEM servers — plain text logs can fill up a volume before overnight compression runs
- Long running database queries — flag anything over 120 seconds and work to improve it
- OS virtual memory swapping — occasional swap is fine, consistent swapping means not enough RAM
- Memory leaks — watch whether a process's memory usage keeps climbing over time
- Report execution time trends — degrading report times signal something is wrong
- High Volume Data Drop monitoring — compare event volumes day by day. If a firewall normally sends 12 million events and suddenly sends 1.5 million, something changed. The book gives an example of a web server being quietly moved to the cloud without telling the SOC, which caused exactly this kind of volume drop.

### Identify and Report IT Operational Issues

This was one of the most interesting parts of the chapter. The book lists real IT problems that were detected through SOC monitoring before the IT team knew about them. It made the SOC role feel more valuable than I had expected.

Real examples from the book:

- A domain controller was failing. Kerberos error events gave it away before authentication started timing out across the WAN.
- A failed software update caused a 1200% increase in VPN authentication failures.
- Disk encryption misconfiguration caused 4.5 million admin logon failures in one hour — 75% of the environment's normal daily authentication volume. This overloaded the domain controller holding the PDC Emulator role.
- MTU size mismatches were causing email delivery to fail for a European office.
- Cached credentials were locking users out when the 60-day password cycle ran.
- Systems reappearing after extended absence and needing domain re-enrollment.
- A developer had built a service using their personal account, then left the company. The service broke when the account was disabled.
- Wireless AP authentication failures in TACACS+ and RADIUS that were blocking real users.
- Software deployment errors visible in application and web server logs.
- Failed backups discovered through log monitoring before a disaster happened.

### Active Threat Hunting

The chapter only briefly covers this as a daily task, pointing to a later section of the book around page 171. The main point here is that threat hunting keeps analysts actively engaged rather than only reacting to alerts.

### Review Security Intelligence Data

Useful sources the book mentions: vendor bulletins, SANS newsletters, AlienVault OTX, security-focused websites, and Twitter feeds from respected security professionals.

The practical advice is that senior staff should identify which sources are worth reading, then subscribe a single shared mailbox to all of them. Every analyst monitors that shared mailbox rather than each person subscribing separately. This prevents good intel from being missed because it only arrived in one person's inbox.

---

### What I took away from this chapter

Before reading this I thought SOC work was mostly watching a screen full of alerts. After reading it I understand it is a structured operational discipline with daily checklists, system health responsibilities, and a real contribution to the broader IT organization.

The parts that stuck with me most:

- Silent data source failures are a genuine risk. You can be blind to an entire part of your environment without knowing it unless you build detection for the gap itself.
- Volumetric changes are indicators, not proof. That distinction matters when making a call on whether something is a real incident.
- The SOC catches operational IT problems that the IT team misses. That makes it valuable well beyond just the security function.
- Alarm triage is a judgment call every time — close it, investigate it, escalate it, or process it yourself. Building that judgment takes time and experience.
